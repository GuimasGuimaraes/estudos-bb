Você já faz isso. Esta apostila entrega o que falta: a terminologia fechada de concurso — as definições, classificações e números que a banca cobra e que ninguém usa no dia a dia.

1. Modelagem de dados

NívelO que defineArtefato
Conceitualo negócio, sem tecnologiaMER / diagrama ER (entidades, atributos, relacionamentos)
Lógicoestrutura no modelo escolhido (relacional), sem SGBD específicotabelas, chaves, cardinalidade
Físicoimplementação realDDL, tipos, índices, tablespaces

Conceitos do MER: entidade (forte × fraca — depende de outra para existir), atributo (simples, composto, multivalorado, derivado, chave), relacionamento e cardinalidade (1:1, 1:N, N:N). Relacionamento N:N vira tabela associativa no modelo lógico. Generalização/especialização = herança.

Chaves: primária (única, não nula), candidata (poderia ser PK), alternativa (candidata não escolhida), composta (2+ colunas), estrangeira (integridade referencial — pode ser nula e repetir), surrogate (artificial, sequencial).

Integridades: de domínio (tipo/valores válidos), de entidade (PK não nula e única), referencial (FK aponta para PK existente) e definida pelo usuário (regras de negócio, CHECK).

FormaExigeElimina

3FNestar na 2FN + nenhum atributo não-chave depender de outro não-chavedependência transitiva
BCNFtodo determinante é chave candidataanomalias residuais da 3FN
4FN / 5FNsem dependência multivalorada / de junçãocasos raros

Desnormalização: introduzir redundância de propósito para desempenho de leitura — típica em DW/OLAP. Troca-se integridade por velocidade.

🪤 Ordem das causas: 1FN=atomicidade · 2FN=parcial · 3FN=transitiva. A banca pergunta "qual FN a tabela viola?" — identifique a dependência, não decore exemplos.

3. SQL — a taxonomia que cai

SublinguagemComandosTransação?
DDL (definição)CREATE, ALTER, DROP, TRUNCATE, RENAMEcommit implícito (Oracle)
DML (manipulação)INSERT, UPDATE, DELETE, (SELECT*)✅ aceita rollback
DCL (controle)GRANT, REVOKE—
TCL/DTL (transação)COMMIT, ROLLBACK, SAVEPOINT—

*SELECT é classificado como DQL por muitas bancas (ou dentro de DML) — se aparecer "DQL", é o SELECT.

Ordem lógica de execução (diferente da ordem escrita!):
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT

Por isso: não se usa apelido do SELECT no WHERE, e agregação no filtro pede HAVING.

Funções de agregação: COUNT, SUM, AVG, MIN, MAX — todas ignoram NULL (exceto COUNT(*), que conta linhas). Toda coluna do SELECT fora de agregação deve estar no GROUP BY.

Operadores de conjunto: UNION (elimina duplicatas), UNION ALL (mantém, mais rápido), INTERSECT, EXCEPT/MINUS.

Views: tabela virtual (consulta armazenada) — simplifica e dá segurança (expõe só colunas permitidas). Materialized view armazena o resultado fisicamente. Índice: em geral B-tree; acelera leitura, onera escrita; bitmap para baixa cardinalidade; clusterizado define a ordem física.

4. Transações e concorrência

ACID: Atomicidade (tudo ou nada) · Consistência (integridade preservada) · Isolamento (transações não se enxergam) · Durabilidade (commit sobrevive a falha).

Nível de isolamentoDirty readNon-repeatable readPhantom
Read Uncommitted✅ ocorre✅✅
Read Committed (padrão Oracle/PG)❌✅✅
Repeatable Read❌❌✅
Serializable❌❌❌
  • Dirty read: leu dado não commitado. Non-repeatable: releu a linha e mudou. Phantom: releu o conjunto e apareceram/sumiram linhas.
  • Deadlock: duas transações esperam recursos uma da outra. O SGBD detecta e aborta uma (a "vítima").
  • Bloqueio otimista (versão/timestamp, assume pouco conflito) × pessimista (lock, assume muito conflito). MVCC: leitores não bloqueiam escritores.

6. NoSQL, DW e Big Data

Teorema CAP: em sistema distribuído, escolha 2 entre Consistência, Availability e Partition tolerance. Como partição é inevitável na rede, o trade-off real é CP × AP. NoSQL costuma seguir BASE (Basically Available, Soft state, Eventual consistency) em vez de ACID.

ModeloExemploUso
Chave-valorRedis, DynamoDBcache, sessão
DocumentoMongoDB, CouchDBJSON flexível
Colunar (wide-column)Cassandra, HBaseescala massiva, séries temporais
GrafoNeo4jrelacionamentos, antifraude

OLTP × OLAP: transacional (muitas escritas curtas, normalizado, dado atual) × analítico (leituras complexas, desnormalizado, histórico).

Data Warehouse (Inmon/Kimball): integrado, orientado a assunto, não volátil e variante no tempo. Modelagem dimensional: fato (métricas) + dimensões. Star schema (dimensões desnormalizadas, mais rápido) × snowflake (dimensões normalizadas). Data mart = recorte departamental. ETL (transforma antes) × ELT (carrega e transforma no destino). Data lake = dado bruto, schema-on-read; lakehouse = híbrido. Os 5 Vs: volume, velocidade, variedade, veracidade, valor.

7. Programação e POO

Pilar da POODefinição de prova
Abstraçãorepresentar o essencial, ocultar detalhes
Encapsulamentoesconder o estado interno; acesso por métodos (getters/setters)
Herançareuso: subclasse estende superclasse (Java: herança múltipla só de interfaces)
Polimorfismomesma interface, comportamentos diferentes

Polimorfismo × sobrecarga: sobrescrita/override (mesma assinatura, subclasse, ligação dinâmica em execução) × sobrecarga/overload (mesmo nome, parâmetros diferentes, ligação estática em compilação). Mudar só o tipo de retorno não é sobrecarga.

Java específico:

  • Classe abstrata × interface: abstrata pode ter estado e construtor, herança simples; interface define contrato, herança múltipla, e desde o Java 8 aceita default e static com corpo (Java 9: métodos privados).
  • Exceções: checked (herdam Exception — obrigam try/catch ou throws: IOException, SQLException) × unchecked (herdam RuntimeException — NullPointerException, ArrayIndexOutOfBounds, ArithmeticException, IllegalArgument). Error (OutOfMemory, StackOverflow) não deve ser tratado.
  • finally executa sempre — inclusive com return no try (exceções: System.exit() ou queda da JVM).
  • String é imutável (métodos retornam nova instância; usa String pool). Mutáveis: StringBuilder (rápido, não sincronizado) e StringBuffer (thread-safe).
  • == compara referência; equals() compara conteúdo. Sobrescreveu equals? Sobrescreva hashCode.
  • Collections: List (ordenada, aceita duplicata) · Set (sem duplicata) · Map (chave→valor) · Queue. ArrayList (acesso O(1), inserção no meio O(n)) × LinkedList (inserção O(1), acesso O(n)). HashMap (sem ordem, permite 1 null na chave) × TreeMap (ordenado, O(log n)) × LinkedHashMap (ordem de inserção). Vector/Hashtable = legados sincronizados.
  • static pertence à classe: método estático não acessa atributo de instância diretamente. final: variável constante, método não sobrescrevível, classe não herdável.
  • Modificadores: private (classe) · default/package (pacote) · protected (pacote + subclasses) · public (todos).
  • JVM: .java → compilador → bytecode (.class) → JVM interpreta/JIT. JDK ⊃ JRE ⊃ JVM. Garbage collector cuida do heap (objetos); a stack guarda variáveis locais e frames.

Árvores: percursos in-ordem (esq→raiz→dir: gera ordem crescente na BST), pré-ordem (raiz→esq→dir) e pós-ordem (esq→dir→raiz). Balanceadas: AVL, rubro-negra. B-tree/B+tree: a base dos índices de SGBD (otimizada para disco). Heap: pai ≥/≤ filhos — base da fila de prioridade.

Grafos: BFS (fila, menor nº de arestas) · DFS (pilha/recursão) · Dijkstra (menor caminho com pesos não negativos) · Kruskal/Prim (árvore geradora mínima).

OrdenaçãoMédioPiorEstável?

Big-O = limite superior (pior caso). Ordem de crescimento: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!).

🪤 Quicksort é o mais rápido na prática mas tem pior caso O(n²); mergesort garante O(n log n) sempre. Busca binária exige vetor ordenado.

9. Engenharia de software

Modelos de processo: Cascata (sequencial, rígido) · Incremental · Espiral (guiado a risco, de Boehm) · RUP (iterativo, 4 fases: concepção, elaboração, construção, transição) · Ágil.

FrameworkEssência
ScrumPapéis: PO (prioriza o Product Backlog, maximiza valor), Scrum Master (facilita, remove impedimentos — não é chefe), Devs (decidem o COMO). Eventos: Sprint (≤1 mês), Planning, Daily (15 min), Review (inspeciona incremento) e Retrospective (melhora o processo). Artefatos: Product Backlog, Sprint Backlog, Incremento — cada um com um compromisso (Meta do Produto, Meta do Sprint, DoD).
KanbanFluxo contínuo, visualização do quadro, limite de WIP, gestão do fluxo, melhoria evolutiva. Sem sprints e sem papéis obrigatórios.
XPProgramação em par, TDD, integração contínua, refatoração, propriedade coletiva do código, cliente presente.
LeanEliminar desperdício, entregar rápido, decidir o mais tarde possível, otimizar o todo.

Manifesto Ágil — os 4 valores (mais à esquerda que à direita): indivíduos e interações > processos e ferramentas · software funcionando > documentação abrangente · colaboração com o cliente > negociação de contratos · responder a mudanças > seguir um plano. (12 princípios.)

🪤 O Manifesto não descarta o lado direito ("há valor nos itens à direita") — alternativa que diz "ágil dispensa documentação" está errada.

Testes

  • Caixa-branca (estrutural, conhece o código: cobertura de linha/condição) × caixa-preta (funcional, só entradas/saídas: partição de equivalência, valor-limite) × caixa-cinza.
  • Níveis: unitário → integração → sistema → aceitação. Regressão garante que o que funcionava continua funcionando; smoke é a verificação mínima.
  • Pirâmide de testes: muitos unitários (base, baratos) → menos de integração → poucos E2E (topo, caros/lentos).
  • TDD: red → green → refactor (teste primeiro, código mínimo, refatora). BDD: Given/When/Then. Erro × defeito × falha: erro humano gera defeito (bug) no código, que pode se manifestar como falha em execução.

UML e padrões

UML — estruturais: classes, objetos, componentes, implantação, pacotes. Comportamentais: casos de uso, sequência, atividades, estados, comunicação.

Relações de classe: associação · agregação (losango vazio — a parte sobrevive ao todo) · composição (losango cheio — a parte morre com o todo) · herança (triângulo) · dependência (tracejado).

Categoria GoFPadrões cobrados

EstruturaisAdapter (compatibiliza interfaces), Facade (simplifica subsistema), Proxy, Decorator, Composite, Bridge, Flyweight

SOLID: Single Responsibility · Open/Closed (aberto a extensão, fechado a modificação) · Liskov (subtipo substitui o tipo base) · Interface Segregation · Dependency Inversion (dependa de abstrações). Complementos: DRY, KISS, YAGNI, coesão alta e acoplamento baixo.

10. Arquitetura e integração

  • Monólito × microsserviços: artefato único × serviços pequenos, autônomos, com deploy independente e banco próprio (database per service). Padrões: API Gateway, Service Discovery, Circuit Breaker, Saga (transação distribuída), CQRS, Event Sourcing.
  • REST: estilo arquitetural sobre HTTP. Restrições: cliente-servidor, stateless, cacheável, interface uniforme, camadas, código sob demanda (opcional). Maturidade (Richardson): 0 → 3 (HATEOAS).
  • Métodos e idempotência: GET (seguro e idempotente), PUT e DELETE (idempotentes), POST não é, PATCH (parcial, não idempotente).
  • Status HTTP: 2xx sucesso (200 OK, 201 Created, 204 No Content) · 3xx redirecionamento (301, 304) · 4xx erro do cliente (400 Bad Request, 401 não autenticado, 403 autenticado sem permissão, 404, 429 Too Many Requests) · 5xx erro do servidor (500, 502, 503).
  • SOAP × REST: SOAP usa XML + WSDL, é protocolo rígido com WS-Security; REST é leve, geralmente JSON.
  • Mensageria: desacopla e é assíncrona. Fila (ponto a ponto, 1 consumidor) × tópico (pub/sub, N assinantes). RabbitMQ (broker AMQP, roteamento rico) × Kafka (log distribuído, alta vazão, retenção e replay). Entrega: at-most-once, at-least-once, exactly-once.
